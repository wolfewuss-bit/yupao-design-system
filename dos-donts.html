Do's & Don'ts

设计规范的最佳实践与常见错误，帮助团队快速理解设计系统的正确使用方式。

颜色使用

Do

  • 使用语义化 Token（如 color/primary/default）引用颜色
  • 同时定义亮色和暗夜两套色值
  • 文字色使用不透明度变体（85/65/45/25）
  • 语义色用于对应场景（红=错误，绿=成功）

Don't

  • 直接硬编码色值（如 color: #0092FF）
  • 只定义亮色主题，忽略暗夜适配
  • 使用纯黑 #000000 作为文字色
  • 混用语义色（如用红色表示成功）

字体排版

Do

  • 使用字号阶梯中定义的字号（10–27px）
  • 行高遵循 font-size + 8px 规则
  • 金额数字使用 DIN Alternate 字体
  • 标题使用 SemiBold/Bold，正文使用 Regular

Don't

  • 使用阶梯外的字号（如 15px、19px）
  • 正文字号小于 12px
  • 标题和正文使用相同字重
  • 在同一页面使用超过 4 种字号

间距与布局

Do

  • 使用 4px 模块化网格的间距值
  • 页面水平边距统一 16px
  • 同层级元素使用相同间距
  • 使用间距 Token 而非硬编码数值

Don't

  • 使用非 4px 倍数的间距（如 5px、7px、15px）
  • 不同页面使用不同的水平边距
  • 间距值随意设定，缺乏规律
  • 忽略底部安全区域

组件使用

Do

  • 按钮文字简洁明确（如"立即投递"、"确认"）
  • 一个页面只有一个主要按钮（Primary）
  • 输入框提供清晰的 placeholder 提示
  • 卡片内容层级清晰（标题 → 描述 → 标签）

Don't

  • 按钮文字过长或含糊（如"点击这里进行下一步操作"）
  • 同一区域放置多个主要按钮
  • 输入框没有任何提示文字
  • 卡片信息堆砌，缺乏层级

命名规范

Do

  • Token 使用 {类别}/{子类别}/{属性} 格式
  • 英文命名，中文注释
  • 使用 kebab-case 命名法
  • 组件命名包含变体和状态

Don't

  • Token 使用中文命名
  • 命名不一致（混用 camelCase 和 kebab-case）
  • 使用缩写导致含义不清
  • 组件命名缺少状态描述

无障碍

Do

  • 触控区域不小于 44×44px
  • 文字对比度达到 WCAG AA 标准（≥4.5:1）
  • 为图片和图标提供替代文本
  • 尊重系统"减少动态效果"设置

Don't

  • 触控区域过小（如 24×24px 的按钮）
  • 浅色背景上使用浅色文字
  • 仅用颜色传达信息（色盲用户无法区分）
  • 忽略屏幕阅读器的语义标记
